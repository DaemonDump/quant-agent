### 🧭 策略开发流程与交互（策略配置/信号/回测）
- **策略类型强制区分与独立模块显示**
  - **问题**：三种策略共用同一“因子权重”模块，导致传统策略与机器学习策略配置混杂。
  - **优化**：

    - 传统策略分别显示各自参数：趋势跟踪（均线/突破窗口/确认天数）、均值回归（回看窗口/阈值/最大持有天数）。
    - 机器学习策略增加模型状态字段（`untrained/training/ready/stale/error`），训练按钮暂置为“待实现”。
  - **涉及文件**：`app/templates/index.html`、`strategy_config/strategy_config.py`

- **策略开发三步流程锁定（防止随意跳转/篡改）**
  - **问题**：用户可随意切换“策略配置→信号→回测”，导致策略在流程中途被改动，结果不可复现。

    - 提供“返回修改策略”入口，解锁会清空信号与回测结果，避免旧结果与新策略混淆。

  - **涉及文件**：`app/templates/index.html`

### 📦 历史数据批量落库（新增脚本）
- **问题**：需要在本地预先落地训练/回测所需的历史数据；手工逐支抓取效率低，且容易触发限速。
- **优化**：
  - 新增脚本 `scripts/fetch_history_batch.py`，支持 `hs300/all/file/list` 四种标的集来源，按日期窗口批量拉取 `daily + daily_basic`，自动写入 SQLite。
  - 支持节流参数 `--sleep` 与分批 `--batch-size`，降低 Tushare 限速风险；支持 `--dry-run` 快速校验。
  - 读取系统内保存的 Token（`settings.tushare_token`），训练阶段不再访问 Tushare，仅“缺口补齐”消耗 Token。
- **典型用法**：
  - 沪深300：`python scripts/fetch_history_batch.py --universe hs300 --start 20220101 --end 20260326`
  - 全 A 股：`python scripts/fetch_history_batch.py --universe all --start 20220101 --end 20260326 --sleep 0.2 --batch-size 500`
  - 列表/文件：`--universe list --symbols "000001.SZ,600519.SH"` 或 `--universe file --file d:\codes.txt`
 - **涉及文件**：`scripts/fetch_history_batch.py`

### 🧠 因子可扩展性（机器学习策略）
- **目标**：后续“增加/修改因子”要低成本、可复现、线上离线一致。
- **优化思路**：
  - **配置驱动**：因子列表（feature_list）与窗口参数通过配置文件统一管理，数据准备/训练/推理都读取同一份配置，避免写死在代码里。
  - **特征版本化**：对 feature_list 计算哈希（feature_list_hash），写入 model_config.json。推理时校验哈希，不一致直接提示“模型需更新/重新训练”，防止线上离线不一致。
  - **统计量随模型走**：训练时生成 feature_stats.json（均值/分位/缺失率/填充策略），推理严格复用，确保缺失值处理一致。
  - **流程隔离**：传统策略参数模块与机器学习因子体系解耦，修改因子不会影响趋势/回归策略。
- **约束**：机器学习策略增删因子必须重新训练；仅调整传统多因子权重/阈值不需要训练。

#### 3. 错误信息去重与流程控制
- **问题**: 在“完整分析”失败时，多个依赖步骤会重复抛出“请先生成信号”的提示，导致错误列表冗余。
- **优化**:
    - **逻辑去重**: 在推送错误信息前增加 `includes` 检查，确保同一分析周期内不出现重复提示。
    - **流程中断**: 在 `runFullAnalysis` 中引入流程控制，若核心信号生成失败，则自动跳过后续的过滤和触发检查步骤。
- **效果**: 错误报告更加精炼，准确指向分析中断的根本原因。

### 🛠️ 系统健壮性与权限修复

**优化日期**: 2026-03-26

#### 1. 解决 Tushare 本地文件写入权限错误 (Errno 13)
- **问题**: 在受限环境下，Tushare 尝试在用户目录写入 `tk.csv` 导致 `Permission denied` 错误，阻断了指数数据的获取。
- **优化**:
    - **Monkey Patch**: 强制重写 `ts.set_token = lambda x: None`，彻底禁用其本地文件持久化逻辑。
    - **显式 Token 传递**: 在实例化 `pro_api(token)` 时直接传入内存中的 Token。
- **效果**: 彻底解决了权限报错问题，确保了在任何系统环境下都能稳定获取行情数据。

#### 2. 数据库访问安全性与健壮性
- **问题**: 数据库辅助函数 `get_setting` 和 `set_setting` 缺乏异常处理，单次失败可能导致连接泄漏。
- **优化**:
    - 增加了 `try-except` 块和 `db.rollback()` 逻辑。
    - 引入了 `current_app.logger` 记录数据库层面的详细错误堆栈。
- **效果**: 提高了后端服务的容错能力，确保了配置项读写的原子性。

  - **需求**：支持选择本地已训练模型并应用到当前策略。
  - **优化**：
    - 新增 `POST /api/ml/import_model`：上传模型文件（支持目录结构），落盘到 `data/tushare/models/imported/<timestamp>/`，自动识别 `model_weights.pkl`/`model.pkl` 并写入策略配置。
    - 前端在“模型状态”卡片新增“选择本地模型/应用模型”。
  - **涉及文件**：`app/routes/ml.py`、`app/templates/index.html`、`strategy_config/strategy_config.py`

- **训练卡住/状态不同步治理（超时自愈 + 可重置 + 可取消）**
  - **问题**：训练线程异常中断或网页刷新后，状态长期停留在 `training`。
  - **优化**：
    - `/api/ml/status`：当训练状态超过阈值未更新时自动转为 `error` 并提示重试/重置。
    - 新增 `POST /api/ml/reset_state`：触发取消标记并重置状态；训练管线检测到取消会尽快退出并清理本次未完成输出目录（如已生成）。
    - 前端增加“重置训练状态”按钮，并在训练中展示 `updated_at`（便于判断是否仍在推进）。
  - **涉及文件**：`app/routes/ml.py`、`aiagent/ml_pipeline.py`、`app/templates/index.html`

- **Windows 写文件并发权限错误（Errno 13）修复**
  - **问题**：训练线程与重置接口并发写入同名 `.tmp` 文件导致 `Permission denied`。
  - **优化**：状态文件/取消标记写入统一改为“唯一 tmp 文件名 + 重试 + 原子替换（os.replace）”，避免争用同一路径。
  - **涉及文件**：`app/routes/ml.py`、`aiagent/ml_pipeline.py`

- **训练进度“卡在 15%”的展示优化**
  - **原因**：特征构建阶段状态写入频率过低 + 进度整数化导致长时间停留在同一百分比。
  - **优化**：提升状态更新频率（按数量与时间双条件），进度改为带小数，同时提示当前处理的股票代码。
  - **涉及文件**：`aiagent/ml_pipeline.py`、`app/templates/index.html`

- **模型导出到任意本地目录（浏览器下载）**
  - **说明**：Web 页面无法让服务端直接写入用户电脑任意目录；改为提供模型压缩包下载，让用户在“另存为”对话框自行选择目录。
  - **优化**：
    - 新增 `GET /api/ml/download_model`：将当前启用模型目录打包为 zip 下载。

  - **涉及文件**：`app/routes/ml.py`、`app/templates/index.html`

### 📦 `data_ingestion/data_collector.py` - 数据采集入库模块

**优化日期**: 2026-03-26

#### 1. 优化数据库连接管理
- **问题**: 在旧实现中，每次数据库操作（如 `store_data`, `get_realtime_data`）都会创建一个新的 `sqlite3.connect()` 连接，并在操作结束后立即关闭。这种方式在高频操作下会产生显著的性能开销。
- **优化**:
    - 在 `RealTimeDataCollector` 类的 `__init__` 方法中初始化一个数据库连接 `self.conn`。
    - 让所有数据库操作方法复用这个持久化的连接。
    - 增加了一个 `close()` 方法，用于在对象生命周期结束时显式关闭连接。
- **效果**: 减少了数据库连接的创建和销毁次数，提高了数据读写的整体性能。

#### 2. 使用批量插入提升数据存储效率
- **问题**: `store_data` 和 `store_history_data` 方法使用 `for` 循环遍历 DataFrame，并逐行执行 `INSERT OR REPLACE` 操作。当数据量较大时，这种逐行插入的方式效率极低。
- **优化**:
    - 使用 `pandas.DataFrame.to_sql()` 方法替换了原有的循环插入逻辑。
    - 设置 `if_exists='append'` 来实现批量追加数据。
- **效果**: 大幅提升了数据入库的速度，尤其是在采集大量历史数据或高频实时数据时，性能改善非常明显。

#### 3. 重构重复的数据存储代码
- **问题**: `store_data` 和 `store_history_data` 方法中的代码逻辑高度相似，存在重复。
- **优化**:
    - 创建了一个通用的私有方法 `_store_data(table_name, data)`，封装了 `to_sql` 的调用逻辑。
    - 让 `store_data` 和 `store_history_data` 在进行数据预处理（如重命名列）后，调用 `_store_data` 方法，从而减少了代码重复。
- **效果**: 提高了代码的可维护性和可读性。

### 📦 `backtest_engine/backtest_engine.py` - 回测与优化引擎

**优化日期**: 2026-03-26

#### 1. 修复归因分析的逻辑错误
- **问题**: `calculate_attribution` 方法试图基于 `profit` 字段进行归因分析，但在 `run_strategy_backtest` 的交易记录中并未计算和保存该字段，导致归因分析功能失效。
- **优化**:
    - 在 `run_strategy_backtest` 方法中，当执行卖出操作时，计算了每笔交易的 `profit`（收益），并将其保存在交易记录中。
    - 修正了 `calculate_attribution` 方法，确保其能正确处理交易记录并进行计算。
- **效果**: 修复了功能性 Bug，使归因分析能够提供准确的交易绩效评估。

#### 2. 优化数据库连接管理
- **问题**: 与 `data_collector.py` 类似，`load_data` 方法在每次调用时都创建新的数据库连接。
- **优化**:
    - 在 `BacktestEngine` 类的 `__init__` 方法中初始化数据库连接，并在 `load_data` 中复用该连接。
- **效果**: 减少了回测过程中不必要的数据库连接开销。

#### 3. 优化策略回测的信号生成循环
- **问题**: `run_strategy_backtest` 的主循环在每个时间步都调用 `strategy_func`，并传入一个不断增长的数据集 (`test_df.iloc[:i+1]`)。这导致了严重的重复计算，极大地拖慢了回测速度。
- **优化**:
    - 在回测循环开始前，通过 `test_df.apply(...)` 一次性为整个测试集生成所有交易信号，并将信号存储在 `test_df` 的一个新列 `signal` 中。
    - 在循环中，直接从该列读取信号，而不是重新计算。
- **效果**: 显著提高了回测性能，将信号生成的时间复杂度从 O(n^2) 降低到接近 O(n)。

#### 4. 改进绩效指标计算
- **问题**: `calculate_metrics` 方法中的绩效指标不够全面。
- **优化**:
    - 增加了 `volatility`（年化波动率）的计算。
    - 修正了 `annual_return`（年化收益率）的计算公式，使其在不同长度的回测周期下都更准确。
- **效果**: 提供了更全面、更精确的策略绩效评估指标。
